# Visual Regression & OG Images

Playwright screenshots from `/preview/[name]` serve as both visual regression baselines and OG social card images. Baselines are committed; `public/og/` is gitignored and regenerated by `pnpm sync:og` (runs automatically as part of `apps/website` `build`).

## Commands

**IMPORTANT:** Always prefix with `CI=1` to prevent Playwright from blocking on the HTML report server when tests fail.

```bash
CI=1 pnpm test:visual                               # Run visual tests against committed baselines
CI=1 pnpm test:visual:build                          # Same, but against production build (no dev overlay noise)
pnpm sync:og                                        # Regenerate OG images locally from baselines

# Single item baseline regeneration
pnpm --filter e2e exec playwright test --project=visual -g <name> --update-snapshots && pnpm sync:og

# Full regeneration
rm -rf apps/e2e/tests/visual.spec.ts-snapshots && \
  pnpm --filter e2e exec playwright test --project=visual --update-snapshots && \
  pnpm sync:og
```

**NEVER** pass `-g`/`--update-snapshots` through `pnpm test:visual -- ...`. pnpm appends trailing args to the shell-chained script, so flags don't reach Playwright. Always call `pnpm --filter e2e exec playwright test ...` directly.

## Pipeline

`visual.spec.ts` screenshots → `apps/e2e/tests/visual.spec.ts-snapshots/<name>.png` + `<name>-dark.png` → `scripts/sync-og.ts` reads light baselines, passes through `scripts/og-overlay.tsx` (satori + Geist fonts, renders title/install-cmd bar) → writes final OG PNGs to `apps/website/public/og/` (gitignored, rebuilt on deploy).

## Key Rules

- Baselines are platform-agnostic (`snapshotPathTemplate` strips project/platform suffix in `apps/e2e/playwright.config.ts`). Light = `<name>.png`, dark = `<name>-dark.png`.
- Opt out a registry item via `meta.visualRegression: false` in its `.meta.json`.
- Independent demos (`demos/*.demo.tsx`) are auto-discovered as visual targets alongside parent meta.json items, provided they don't collide with an existing meta.json name.
- Collection-only items (no `.tsx`/`.ts` — only meta + `registryDependencies`) auto-excluded by `apps/e2e/tests/registry-items.ts`.
- Test asserts no `pageerror` and no Next.js error dialog (`[data-nextjs-dialog]`) before screenshot — prefer `test:visual:build` in CI.
- **NEVER** edit `public/og/*.png` directly — file is gitignored + regenerated at build time.
- **Gotcha:** `--update-snapshots` only rewrites when diff exceeds `maxDiffPixelRatio: 0.01` (1%). For small content changes, delete baselines first to force a full rewrite.

## Pre-screenshot Interactions (`visual-setups.ts`)

Some registries need user interaction (click, hover) before screenshotting to show their meaningful state. These setups live in `apps/e2e/tests/visual-setups.ts` — a static map keyed by registry name.

### How it works

1. `visual.spec.ts` navigates to `/preview/<name>`, waits for fonts + networkidle.
2. Looks up `visualSetups[item.name]`. If a setup exists, calls it with `{ page, theme }`.
3. Takes screenshot via `toHaveScreenshot`.

### When to add a setup

Add a setup when the default rendered state is empty/unhelpful for OG/visual preview:

- **Select/Menu components** — click to open the dropdown so options are visible.
- **Tooltip components** — hover to show the tooltip.
- **Popup/Flyout components** — hover or click triggers to expand nested content.
- **Collapsible/Accordion** — click to expand a section.

### How to add a setup

1. Open `apps/e2e/tests/visual-setups.ts`.
2. Add an entry keyed by registry name with an `async ({ page, theme }) => { ... }` function.
3. Delete old baselines and regenerate:
   ```bash
   rm apps/e2e/tests/visual.spec.ts-snapshots/<name>*.png
   pnpm --filter e2e exec playwright test --project=visual -g <name> --update-snapshots && pnpm sync:og
   ```
4. Verify the OG at `apps/website/public/og/<name>.png`.

### Finding selectors

Use `/agent-browser` to explore the live preview:

```bash
agent-browser open http://localhost:4418/preview/<name>
agent-browser snapshot -i -C       # list interactive elements with refs
agent-browser hover @e1            # try hover
agent-browser click @e2            # try click
agent-browser snapshot -i -C       # re-snapshot to see what appeared
agent-browser screenshot /tmp/check.png
```

### Common patterns

**MUI Select / Menu** — click trigger, wait for `.MuiMenu-paper`:

```ts
'my-select': ({ page }) =>
  openMuiMenu({ page, trigger: '[role="combobox"]' }),
```

**MUI IconButton with Menu** — click `button[aria-haspopup="true"]`:

```ts
'my-icon-menu': ({ page }) =>
  openMuiMenu({ page, trigger: 'button[aria-haspopup="true"]' }),
```

**MUI Tooltip** — hover trigger, wait for `.MuiTooltip-tooltip`:

```ts
'my-tooltip': async ({ page }) => {
  await page.getByRole('button', { name: 'Label' }).hover();
  await page.locator('.MuiTooltip-tooltip').first().waitFor({ state: 'visible' });
},
```

**Base UI NavigationMenu (nested popups)** — sequential hovers:

```ts
'my-nav': async ({ page }) => {
  await page.getByRole('button', { name: 'Parent' }).hover();
  const child = page.getByRole('button', { name: 'Child' });
  await child.waitFor({ state: 'visible' });
  await child.hover();
  await page.getByRole('link', { name: 'Leaf' }).waitFor({ state: 'visible' });
},
```

### Tips

- Always `waitFor({ state: 'visible' })` after interaction — the popup/tooltip may animate in.
- For hover-based popups, Playwright keeps the cursor on the element until the next mouse action, so `toHaveScreenshot` captures the open state correctly.
- `theme` is passed to setup in case behavior differs by color scheme (rarely needed).
- Use `openMuiMenu` helper for anything that opens a `Menu` or `Select` dropdown.
